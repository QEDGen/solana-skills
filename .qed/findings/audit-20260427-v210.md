## Remaining open items (spec-shape, not codegen)

### multisig::approve/reject — need member list + voted bitmap

The spec models membership as a count (`member_count`) rather than a list (`members : Map[N] Pubkey`). The signer can claim any `member_index < member_count` without proving they're at that index. Same `member_index` can be re-voted by different signers (no bitmap).

**Spec edit required:**

```diff
 type State
   | Active of {
       creator         : Pubkey,
       threshold       : U8,
       member_count    : U8,
+      members         : Map[32] Pubkey,
+      voted_bitmap    : U32,
       approval_count  : U8,
       rejection_count : U8,
     }
```

Plus `requires state.members[member_index] == approver` and `requires (state.voted_bitmap >> member_index) & 1 == 0` on `approve` and `reject`.

### lending::liquidate — need explicit health-threshold check

The spec doesn't gate liquidation behind a health check. Adding a `requires` clause that compares loan amount to collateral closes this.

**Spec edit required:**

```diff
 handler liquidate (...) : Loan.Active -> Loan.Liquidated {
   ...
+  requires state.amount > state.collateral else AccountHealthy
   effect { ... }
 }
```

### `todo!()` placeholders (12 handlers)

Each non-trivial handler bottoms out in `todo!("fill non-mechanical effects, events, transfers, calls")`. These are codegen scaffolding — user fills before deploying. Not an audit finding against the spec; they're placeholders the user is meant to fill, and panicking at the panic site is the right "I'm not deployed" signal.

## Original findings (kept for reference)

[All sections below this point are the initial audit report — preserved verbatim. The closure rows in the table at the top mark which ones are now resolved.]

### [CRIT] percolator — every handler — missing authority binding [✓ closed by R25]

State struct stored `pub authority: Address` but no constraint tied the signer to it. Now every percolator handler emits `#[account(mut, has_one = authority)] pub vault: …`.

### [CRIT] multisig::propose — proposal-erasure governance attack [✓ closed by R26]

Status discriminator was decoration only. Now `propose` requires `status == Active` and writes `status := HasProposal`, so a second call from `HasProposal` fails with `InvalidLifecycle` instead of zeroing the votes.

### [HIGH] multisig::approve/reject — no signer↔member binding [OPEN — spec-shape]

See above. R25 doesn't apply because the spec doesn't yet model a member list. Spec edit required.

### [HIGH] multisig::remove_member — any signer can remove members [✓ closed by R25]

Now emits `#[account(..., has_one = creator)]`.

### [HIGH] lending — pool_vault not bound to pool [✓ closed by R27]

R27 emits a runtime `*ctx.pool_vault.owner() == *ctx.pool.address()` check on every non-init handler with `authority pool` declared.

### [HIGH] lending::liquidate — liquidator unrestricted [OPEN — spec-shape]

Spec edit required (health-threshold `requires`).

### [MED] R13 trade-off — non-init seed verification suppressed [✓ closed by R28]

R28 emits a runtime `verify_program_address` syscall using the stored bump on every R13-suppressed account. Cost: ~544 CU first-try.

### [INFO] no `has_one` constraints anywhere [✓ closed by R25]

Now emitted system-wide whenever the spec's `auth X` clause matches a state field.

### [INFO] spec status discriminator unused at runtime [✓ closed by R26]

Now read and written by guards.rs.
